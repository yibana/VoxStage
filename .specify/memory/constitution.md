# VoxStage Constitution

## Core Principles

### I. 剧本优先 & 语法兼容

VoxStage 的核心资产是「剧本」及其领域语言（VOX Script）。任何新特性都必须首先考虑对剧本语法、可读性和向后兼容性的影响。  
剧本编辑必须始终支持「图形编辑模式」与「Code 模式」双向无损转换，同一行语句在任何模式下都能准确映射与高亮。  
循环、条件、BGM、表达式等语法特性在引擎、runner 与 GUI 中的行为必须保持一致，不允许出现「编辑器能写但 engine 无法运行」或「engine 支持但前端无法呈现」的割裂。

### II. Engine / Runner / GUI 分层解耦

工程结构必须保持清晰分层：

- **vox-engine**：只关心脚本解析、执行模型、缓存策略等核心逻辑。
- **runner/CLI**：负责命令行入口、与文件系统和环境的交互。
- **GUI（Tauri + Vue3）**：只负责交互和可视化，不嵌入业务规则。

前端不得直接绕过 Tauri 命令操作底层资源；所有跨层交互都通过明确的 API/事件桥接（如 `script-progress`、`script-finished` 等）。  
任何修改如果会跨层影响，必须在 Spec 中写清楚「引擎接口 → Tauri 命令 → 前端状态/高亮」的完整链路。

### III. 可观测性与可调试性（行号、事件、日志）

系统必须内建良好的可观测性：

- 所有脚本执行进度都需带有**行号/来源索引**，以便前端高亮和错误定位。
- 事件 ID 与脚本结构（循环、条件）保持稳定映射，同一条语句的多次执行应能在前端被准确识别。
- 关键路径（TTS 调用、缓存命中/未命中、BGM 播放、IO 错误）必须有结构化日志或清晰的错误提示。

在新增功能（如缓存、循环运行、窗口记忆等）时，要首先考虑「如何调试」：  
例如是否能在前端看到进度、在日志中看到失败原因，而不是只出现「运行脚本时前端会卡住」之类的模糊现象。

### IV. 可扩展 Provider 与配置驱动

VoxStage 支持多种语音提供方与配置（如角色、language、ref_audio_path、自定义参数等）。  
所有与 Provider 相关的行为必须遵守以下原则：

- **配置驱动**：使用统一的配置结构（App 配置 / 角色配置 / Provider 参数），避免把 Provider 细节硬编码在脚本中。
- **前端专用控件**：对常见字段（如 language、ref_audio_path、风格参数等）通过专用 UI 控件管理，而不是靠用户手写 Key/Value。
- **向后兼容**：增加字段时不破坏已有脚本，缺省值和默认行为必须定义清楚。

当新增 Provider 或参数时，必须同时更新：

- 对应的配置结构与解析逻辑；
- 前端的配置编辑 UI；
- 帮助文档/README 中的说明。

### V. 简单、安全与资源控制

优先保持实现简单可维护，必要功能优先于复杂度堆叠：

- 缓存策略必须有限制（如最大条目数 + LRU 淘汰），避免内存/磁盘无限增长。
- GUI 的窗口尺寸和位置记忆功能必须考虑多屏幕/分辨率变化，保证不会把窗口放在屏幕外或过大。
- 脚本运行时的循环和并发行为必须有明确的终止条件和中断机制（暂停/中断/循环开关等），避免误操作导致资源长期占用。
- 错误信息要聚焦用户能理解的层面（例如脚本行号、语法错误位置、缓存失败原因），而不是只给出底层 panic 或框架报错。

在做设计决策时，如果存在「稍复杂但更安全」与「简单但潜在有坑」的选择，应优先选择前者，但同时保持实现与文档清晰。

## 架构与技术栈约束

VoxStage 的主要架构与技术栈约束如下：

- **语言与框架**
  - 后端：Rust（Cargo Workspace），模块化拆分 engine / runner / GUI Tauri 后端。
  - 桌面 GUI：Tauri 2 + Vue 3 + Vite，使用 pnpm 作为包管理器。
- **GUI 构建与打包**
  - 开发：`pnpm run dev:tauri`，前端使用 Vite 开发服务器。
  - 发行版：`pnpm run tauri build` 必须始终在 GUI 子项目中执行，让前端资源自动打包进 `.exe`。
  - 仅打包 exe：通过在 `tauri.conf.json` 中设置 `bundle.active = false`，只生成可执行文件而不生成安装包。
- **脚本与资源**
  - 剧本文件使用 VoxStage 自定义语法（支持循环、条件、BGM、表达式等），必须保证解析与回写是可逆的。
  - 音频缓存等资源相关功能需要统一的配置入口（GUI App Tab / 配置文件），和明确的默认策略（是否开启缓存、最大缓存条目、缓存位置等）。

## 开发流程与规范

开发过程应遵循以下工作流：

- **Spec‑Kit 驱动**
  - 新功能或较大改动，先编写或更新对应的 Spec/Plan 文档（`.specify` 下），再开始编码。
  - Spec 中必须明确：目标、输入输出、边界条件、错误处理、与现有模块的耦合关系。
- **任务拆分**
  - 使用 Speckit 的 tasks 模板，将功能拆分为若干小任务（如「GUI 表达式输入增强」「engine BGM 语法支持」「文档与帮助更新」），并在实现时逐个勾选完成。
- **文档与帮助**
  - 所有用户可见的功能（包括 GUI 交互、脚本语法、缓存策略、Release 步骤），必须在根 `README.md` 和 GUI 中的帮助页面中有对应说明。
  - 发行版发布前，需要检查 README 与帮助页面是否与当前实现一致（尤其是脚本语法和配置项）。
- **版本与发布**
  - 使用语义化版本：`MAJOR.MINOR.PATCH`。
    - MAJOR：脚本语法或行为有不兼容改动。
    - MINOR：新增向后兼容的功能（如缓存、表达式辅助输入、GUI 增强）。
    - PATCH：修复 bug、优化体验、不改变对外契约。
  - 发布新版本时：
    - 更新 Cargo 相关版本号（如有需要）。
    - 更新 `README.md` 和 GUI 帮助页面。
    - 打包 CLI 与 GUI 二进制，上传到 GitHub Releases，并在 Release 说明中列出变更摘要。

## Governance

- 本宪章是 VoxStage 项目的最高级工程规范，对代码结构、功能设计和文档有约束力。
- 任何违反本宪章的改动必须在 Spec 中明确说明理由，并在 Review 时进行讨论。
- 修改宪章时：
  - 必须通过 Speckit 的 constitution 命令更新 `.specify/memory/constitution.md`。
  - 根据改动性质判断版本号变更（MAJOR/MINOR/PATCH），并在顶部同步「版本变更说明」。
  - 检查并同步更新相关模板（plan/spec/tasks）和用户文档（README、GUI 帮助页）。
- 所有 PR / 提交在 Review 时，都应至少回答三个问题：
  1. 是否遵守了分层与职责划分（engine/runner/GUI）？
  2. 是否保证了脚本语法的可逆性与向后兼容？
  3. 是否更新了相应的文档/帮助或留下明确的 TODO？
